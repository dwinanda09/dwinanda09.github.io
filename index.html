The classic approach

What is REST?

REST (Representational State Transfer) is an architectural style for APIs. Think of it like ordering food at a restaurant — you say what you want in plain language and get a response back.

  • Uses HTTP/1.1 verbs: GET, POST, PUT, DELETE
  • Data in JSON — human-readable text
  • Stateless — each request is independent
  • Works everywhere: any language, any browser
🌍 Used by: Twitter API, GitHub API, Stripe, almost every public API you've seen
# GET a user — REST style GET /users/42 Accept: application/json # Response (JSON) { "id": 42, "name": "Dwi Nanda", "role": "engineer" }
Google's answer to fast communication

What is gRPC?

gRPC lets your code call a function on another machine as if it were local. Like calling a friend directly by phone, not via email.

2015
Open-sourced by Google
Protocol
Built on HTTP/2 — multiplexed, binary frames
Format
Protocol Buffers — binary serialization
Schema
Strongly typed .proto contract file
⚡ Used by: Google internal services, Netflix, Dropbox, Slack microservices
// user.proto definition service UserService { rpc GetUser(UserRequest) returns (UserResponse); } message UserRequest { int32 id = 1; } message UserResponse { int32 id = 1; string name = 2; string role = 3; }

The hidden superpower

gRPC Streaming

gRPC supports 4 communication patterns. REST only supports 1.

1. Unary — like REST
One request → one response. Simple.
2. Server Streaming
Client sends once, server streams back data.
Example: live stock prices, log tailing
3. Client Streaming
Client streams data, server replies once.
Example: file upload, sensor telemetry
4. Bidirectional Streaming
Both sides stream simultaneously. Real-time!
Example: chat app, multiplayer game sync
💡 REST can fake streaming with SSE or WebSockets — but it's bolted on. gRPC streaming is native and first-class.

Use case guide

When to pick gRPC

  • Internal microservice-to-microservice calls
  • High-throughput, low-latency requirements
  • Real-time bidirectional streaming
  • Mobile clients on slow/metered connections
  • Polyglot environments needing strong contracts
Real example:
Netflix uses gRPC for 1,000+ internal services communicating millions of times per second.
🏭
Microservices
Services calling each other thousands of times/sec
📡
Streaming data
IoT telemetry, live dashboards, chat
📱
Mobile backends
Smaller payloads = faster on 4G
🔒
Strong contracts
Protobuf enforces schema — no surprises
Use case guide

When to pick REST

  • Public-facing APIs consumed by third parties
  • Browser-based frontends (SPA, SSR)
  • Simple CRUD services with low traffic
  • Teams new to backend APIs
  • When debuggability & curl-ability matters
Real example:
Stripe, GitHub, Twilio — all public REST APIs. Explore them directly in Postman or the browser.

Quick guide

Decision Flowchart

Is this a public API (consumed by external developers)?
→ Yes: Use REST  |  → No: continue ↓
Do you need real-time streaming (chat, live data, telemetry)?
→ Yes: Use gRPC  |  → No: continue ↓
Is this internal service-to-service at high volume?
→ Yes: Use gRPC  |  → No: continue ↓
Is debugging ease more important than raw performance?
→ Yes: Use REST  |  → No: gRPC is fine
When in doubt? Start with REST. Migrate hot paths to gRPC when performance actually becomes a problem. Don't premature-optimize!

The real answer: "It depends" — but now you know exactly what it depends on.
  • gRPC is faster: HTTP/2 + binary
  • REST is simpler: works everywhere
  • Both can coexist in the same system
  • Performance matters at scale
  • gRPC shines inside data centers
  • REST shines at the edge (public APIs)
  • Pick based on your actual use case
