The classic approach

What is REST?

REST (Representational State Transfer) is an architectural style for APIs. Think of it like ordering food at a restaurant — you say what you want in plain language and get a response back.

  • Uses HTTP/1.1 verbs: GET, POST, PUT, DELETE
  • Data in JSON — human-readable text
  • Stateless — each request is independent
  • Works everywhere: any language, any browser
🌍 Used by: Twitter API, GitHub API, Stripe, almost every public API you've seen
# GET a user — REST style GET /users/42 Accept: application/json # Response (JSON) { "id": 42, "name": "Dwi Nanda", "role": "engineer" }
Google's answer to fast communication

What is gRPC?

gRPC lets your code call a function on another machine as if it were local. Like calling a friend directly by phone, not via email.

2015
Open-sourced by Google
Protocol
Built on HTTP/2 — multiplexed, binary frames
Format
Protocol Buffers — binary serialization
Schema
Strongly typed .proto contract file
⚡ Used by: Google internal services, Netflix, Dropbox, Slack microservices
// user.proto definition service UserService { rpc GetUser(UserRequest) returns (UserResponse); } message UserRequest { int32 id = 1; } message UserResponse { int32 id = 1; string name = 2; string role = 3; }

REST / HTTP 1.1 — one TCP connection = one request at a time. While req #1 travels, req #2 & #3 must wait in line (head-of-line blocking). Large JSON payloads add extra round-trip overhead.
gRPC / HTTP 2 — one TCP connection carries multiple streams simultaneously (multiplexing). Tiny binary Protobuf frames travel in parallel — no waiting, no blocking, 4× faster in this example.
The hidden superpower

gRPC Streaming

gRPC supports 4 communication patterns. REST only supports 1.

1. Unary — like REST
One request → one response. Simple.
2. Server Streaming
Client sends once, server streams back data.
Example: live stock prices, log tailing
3. Client Streaming
Client streams data, server replies once.
Example: file upload, sensor telemetry
4. Bidirectional Streaming
Both sides stream simultaneously. Real-time!
Example: chat app, multiplayer game sync
💡 REST can fake streaming with SSE or WebSockets — but it's bolted on. gRPC streaming is native and first-class.

Use case guide

When to pick gRPC

  • Internal microservice-to-microservice calls
  • High-throughput, low-latency requirements
  • Real-time bidirectional streaming
  • Mobile clients on slow/metered connections
  • Polyglot environments needing strong contracts

Quick guide

Decision Flowchart

Is this a public API (consumed by external developers)?
→ Yes: Use REST  |  → No: continue ↓
Do you need real-time streaming (chat, live data, telemetry)?
→ Yes: Use gRPC  |  → No: continue ↓
Is this internal service-to-service at high volume?
→ Yes: Use gRPC  |  → No: continue ↓
Is debugging ease more important than raw performance?
→ Yes: Use REST  |  → No: gRPC is fine
When in doubt? Start with REST. Migrate hot paths to gRPC when performance actually becomes a problem. Don't premature-optimize!

The real answer: "It depends" — but now you know exactly what it depends on.
  • gRPC is faster: HTTP/2 + binary
  • REST is simpler: works everywhere

  • REST shines at the edge (public APIs)

Design tradeoffs

Why this shape?

Parsing offloaded to S3-event Lambda
Upload API doesn't block on Excel parsing. File lands in S3, Lambda picks up the event — API returns 202 in milliseconds so the dashboard never times out on a 20K-row file.
SQS as elastic backpressure
Lambda dumps the full row bundle into SQS. Consumer cluster reads at its own pace; traffic spikes turn into queue depth, not failed requests — parser throughput stays decoupled from consumer throughput.
5 fan-out goroutines per message
Each row triggers concurrent work — product-service for premium calc, agent-service for agent resolution, DB writes, PDF generation. Per-row I/O parallelism collapses serial wait time.
Stateless consumer cluster
Any instance can pick up any message. Cluster scales horizontally on SQS depth without sticky routing — no ops intervention during the upload peak.

Design tradeoffs

Why this shape?

Scheduler-driven polling, never user-blocking
User-facing API returns 201 in ms after submit. Vendor latency (30s–3min) absorbed by Scheduler ticks polling per-engine queues. FE polls our DB, never the vendor.

Terminology Legend

xG · Expected Goals
Model estimate of goal output based on chance quality, not actual scored. Pipeline uses Elo win-prob × tournament baseline (2.5 group / 2.1 knockout) as prospective xG.
Poisson · draw probability
Statistical distribution assuming goal independence. Yields draw likelihood from expected goals — known to undercount defensive group-stage clustering.
Elo · team rating
Zero-sum team-strength number; higher = stronger. Rating difference maps to win probability via a logistic curve. Backbone of the baseline model.
H2H · Head-to-Head
Historical record between two nations, split by World-Cup-only vs full all-time. Feeds LLM prompt as prior context.
gpm · Goals per Match
Recent-form scoring rate over the team's last 10 matches. Flags low-scoring form ("park the bus" risk).
RPM · Requests per Minute
LLM API rate limit. Sliding-window enforcer keeps calls under free-tier ceiling (13/60s vs 15 hard limit).

Design tradeoffs

Why this shape?

Multi-LLM consensus, score-truthful averaging
Naïve averaging of {1-0, 0-1} yields 0-0/1-1 — a phantom nobody actually predicted. Filter scores to only those matching consensus winner before averaging.
Sliding-window RPM limiter (deque of timestamps)
Free-tier quota = 15 RPM; we enforce 13. deque[float] of request timestamps, prune entries older than 60s, sleep until oldest expires if full. Zero external state, thread-safe with a single lock.
Knockout-aware pipeline branch
Group: xG base 2.5, standard Poisson. Knockouts: xG base 2.1, draw damping 0.7, favorite-edge boost. Same core pipeline — a single is_knockout flag flows through prompt + scoring.
Iterative back-analysis before next round
After group stage: back-analysis docs surfaced failure modes (draw underprediction, LLM verdict inversion, dead-rubber blindness). Ranked upgrades fed forward into R32 pipeline instead of full rewrite.

Design tradeoffs

Why this shape?

Two-phase inquiry → payment
Matches Telkomsel's protocol. Prevents over-billing on client retries — inquiry is safely idempotent, payment is confirmed only once.
mTLS upstream, signed-cipher downstream
Different trust boundaries get different controls. Tokopedia's standard requires payload-level encryption + signing; Telkomsel uses mTLS at transport layer.
Settlement as separate event stream
Reconciliation runs out-of-band, never blocks the user-facing request path. Lossy network days don't degrade transaction latency.
Keys live in Vault, never inlined
Rotation handled centrally. Code reviews can't accidentally introduce a leaked key — there's nothing to leak in the repo.
